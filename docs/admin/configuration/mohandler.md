# HTTP MO Handler

## Overview

The **HTTP MO Handler** in iTextPro is used to receive and process incoming **MO (Mobile Originated)** messages from telecom operators or gateway vendors. The handler acts as an API endpoint that accepts MO requests and forwards the received data to the platform for further routing and processing.

The HTTP MO Handler configuration defines:

- Communication method
- Channel details
- Payload mapping
- Security restrictions
- Endpoint generation

!!! warning "Prerequisite"
    This configuration is **mandatory** before creating MO Routing Rules.

---

## Navigation Path

`Configuration` ➔ `HTTP MO Handlers` ➔ `Add New Handler`.

## Handler Configuration Parameters

The following parameters must be configured while creating an HTTP MO Handler.

### 1] Handler Name

The **Handler Name** is used to uniquely identify the MO Handler configuration within the platform.

This name is internally used for:

- Routing configuration
- Handler selection
- MO traffic mapping
- Administration and troubleshooting

!!! example
    ```
    MO_HANDLER_INDIA_01
    ```

---

### 2] Channel Type

The **Channel Type** defines the type of telecom number associated with the incoming MO traffic.

**Supported Channel Types:**

| Type | Description |
|------|-------------|
| **Long Code** | A standard mobile number used for two-way messaging communication. |
| **Short Code** | A short numeric code generally used for enterprise campaigns, voting systems, subscriptions, or promotional services. |

The selected channel type should match the operator or vendor configuration.

![Manage MO Handler — full configuration form](images/mohandler-02-add-new.png)

---

### 3] Channel Number

The **Channel Number** represents the actual destination number on which MO traffic will be received.

!!! example
    ```
    567890
    ```

This number must be configured **exactly as provided** by the telecom operator or gateway vendor.

---

### 4] Local Endpoint

The **Local Endpoint** is automatically generated by iTextPro once the handler configuration is created.

This endpoint acts as the receiving URL for incoming HTTP MO requests.

The generated endpoint URL is generally shared with:

- SMS Gateway Vendors
- Telecom Operators
- Aggregators
- Third-party Platforms

The vendor uses this endpoint to push incoming MO messages into the iTextPro platform.

**Example Flow:**

```
Operator/Vendor → Local Endpoint → iTextPro Processing
```

---

### 5] Whitelist IP

The **Whitelist IP** section is used to secure the MO endpoint by restricting access only to authorized IP addresses.

Only requests received from configured IP addresses will be accepted by the platform.

**Purpose:**

- Prevent unauthorized access
- Enhance API security
- Restrict unknown traffic
- Protect MO endpoint from misuse

!!! example
    ```
    192.168.10.20
    ```

!!! tip
    Multiple IPs can be configured as per operational requirements.

---

### 6] Method

The **Method** defines the HTTP communication method used by the vendor while sending MO requests.

**Supported Methods:** `GET`, `POST`.

#### GET Method

In the GET method:

- Parameters are transmitted within the URL.
- Data is sent as query parameters.
- Suitable for lightweight integrations.

!!! example
    ```
    https://domain.com/mo?originator=9876543210&destination=567890&message=TEST
    ```

#### POST Method

In the POST method:

- Parameters are transmitted within the HTTP request body.
- Supports structured and larger payloads.
- Commonly used for modern API integrations.

**Benefits:**

- Better security
- Cleaner request structure
- Supports JSON / XML payloads
- Suitable for complex integrations

---

## Payload Configuration

The **Payload** section defines how incoming request parameters will be mapped within iTextPro.

Correct payload mapping is **mandatory** for successful MO processing. Configure the payload parameters exactly as mentioned below:

| Platform Parameter | Vendor Parameter |
|--------------------|------------------|
| **Originator** | `originator` |
| **Destination** | `destination` |
| **Message** | `message` |

### Payload Parameter Description

#### Originator

The **Originator** parameter represents the sender mobile number from which the MO message is received.

!!! example
    ```
    9876543210
    ```

#### Destination

The **Destination** parameter represents the **Short Code** or **Long Code** number on which the MO message was sent.

!!! example
    ```
    567890
    ```

#### Message

The **Message** parameter represents the actual text content submitted by the end user.

!!! example
    ```
    ASKRK BALANCE
    ```

---

## Save Handler Configuration

After completing all configurations:

1. Verify all configured details.
2. Click on **Save**.

The HTTP MO Handler is now successfully configured and ready to receive MO traffic.

!!! tip "Next Step"
    Once the handler is saved, proceed to create an **MO Routing Rule** to define how the incoming MO traffic will be filtered and delivered to end users.
