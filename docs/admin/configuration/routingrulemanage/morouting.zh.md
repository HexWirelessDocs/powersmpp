# MO 运行规则

## 概览

**MO 运行规则** iTextPro中用于定义如何在平台内识别、过滤和路由收到的MO消息。

路线规则决定:

- 哪些输入的MO流量应该处理
- 哪个关键字应触发路由
- 哪个用户应该接收MO流量
- 用于交付的接口类型

MO 运行规则与配置 **HTTP MO 处理器**。 。 。 。

## MO 规则参数

在创建路由规则时必须配置以下参数.

## 通用参数

### 1] 规则名称

这个 **规则名称** 唯一在平台内识别MO Routing Rule.

此名称用于:

- 规则管理
- 交通监测
- 行政部门
- 解决问题

!!! example
    ```
    MO_ROUTE_KEYWORD_01
    ```

---

### 2] 寿命

这个 **终身** 参数定义了路由规则的有效性期限。

**用法 :**

- 可用于临时活动
- 支持基于时间的MO路由
- 对有限时间服务有用

!!! tip
 如果不需要过期,此字段可留为空白.

---

## 网关界面配置

### 处理器

这个 **处理器** 字段用于选择先前在平台中配置的 HTTP MO Handler。

此处理器将处理所有匹配路由条件的MO请求. 处理器将被使用,以防供应商发送MO与 **HTTP 连接**。 。 。 。

### 网关

如果供应商支持 **SMPP 软件** 发送 MO 点击的协议,然后在 MO 运行规则的创建过程中,管理员需要选择 **网关** 并选择正确的网关,以便从正确的网关获取点击。

**目的:**

- 将 MO 流量与正确的端点连接
- 联系人与进入频道的路线
- 启用信件处理工作流程

## 运行条件

路线条件界定 **过滤逻辑** 运入的MO流量。 该平台在处理或引导MO请求之前对这些条件进行评估.

### 1] 发端人条件

这个 **发件人** 条件根据发送者移动号定义过滤。

**示例配置 :** <span data-ph="0"></span>

选择 **任意** 允许来自所有发送者的MO消息。 如果需要,也可以配置特定的发送过滤器.

---

### 2] 目的地条件

这个 **目标** 条件定义接收的短码或长码号.

| 外地 | 数值 |
|-------|-------|
| **条件类型** | <span data-ph="0"></span> |
| **示例** | <span data-ph="0"></span> |

只有收到的MO消息在所配置的目的地号码上收到,路由规则才会被触发.

---

### 3] 信件条件

这个 **消息** 条件定义关键字匹配标准。

| 外地 | 数值 |
|-------|-------|
| **条件类型** | <span data-ph="0"></span> |
| **示例关键词** | <span data-ph="0"></span> |

只有收到的消息以所配置的关键词开头,路由规则才会触发.

!!! example
 对于收到的消息 <span data-ph="0"></span>,从消息开头 <span data-ph="1"></span>,路由规则将处理MO请求.

## 用户和端点映射

### 1] 用户

选择 **用户账户** MO的流量应该被绘制并交付给它.

此映射可确保正确的用户收到收到的MO消息.

### 2] 用户界面类型

这个 **用户界面类型** 定义 MO 消息在路由后应如何传输。

**支持的接口类型 :**

| 类型 | 说明 |
|------|-------------|
| **未选中** | 不应用特定接口路由 。 |
| **无害环境管理** | 通过SMPP连通来引导MO交通. 一般在用户通过SMPP协议连接时使用. |
| **网页** | 将MO流量路由到一个外部的HTTP API端点. 一般用于CRM集成,第三方应用程序,网络处理系统,以及API驱动的工作流程. |

---

## 保存规则

配置所有路由参数后 :

1. 校验路线条件.
2. 验证关键字配置 。
3. 点击 **保存**。 。 。 。

MO Routing Rule现在已成功配置并活跃于MO流量处理.

!!! tip "核查"
 保存规则后,发送符合所配置条件的测试MO消息(允许发送,校正目的地号,消息从所配置的关键词开始),并通过检查用户的MO收件箱或webhook端点日志确认规则起火.
